# Phase 5 — Ops: HTTPS, Backups, Purge, Config, CI, README — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Make the app operable and defensible: a guarded "purge all candidate data" (right to erasure), admin config (prep_window_days / retention_date / qa_sla_text), automated Postgres backups with documented restore, the Docker image build added to CI, and a complete README. (HTTPS/Caddy is already configured from Phase 1 — this phase verifies and documents it.)

**Architecture:** A `purge` admin endpoint deletes candidate data in FK-safe order inside one transaction, requiring an exact typed confirmation phrase; it removes candidates + their bookings/questions/download_events + candidate-attributable audit rows, while keeping the admin account, config, and admin-actor audit rows. A `config` admin endpoint reads/writes the `config` table with validation. Backups are a small `pg_dump` script run on a cron sidecar container, retained locally with rotation; restore is documented. CI gains an image-build job. The frontend adds a config/ops admin page with the guarded purge control. The README ties together setup, env, manual admin steps, DNS, backup/restore, and the purge.

**Tech Stack:** Same as Phases 1–4. The backup sidecar uses the stock `postgres:16` image's `pg_dump` + a cron loop (no new app deps).

## Global Constraints

- **Right to erasure (spec §3.7 / §6 Phase 5):** ONE guarded admin action purges ALL candidate data — `candidate`, `booking`, `question`, `download_event`, and candidate-attributable `audit_log` rows (rows whose `actor` is a candidate id). KEEP: the `admin` account, the `config` table, and admin-actor audit rows (the process/defensibility record; `candidate_id` is a system id, not personal data per §3 — only `first_name` is personal data and it lives on `candidate`, which is deleted). Require an EXACT typed confirmation phrase. The purge is manual (admin-triggered), never automatic.
- **`retention_date` is a clearly-marked placeholder, unset (null) by default** (spec §4); displayed as a reminder only — it does NOT trigger automatic deletion.
- **No secrets** in repo/logs; backups contain DB data (incl. encrypted api keys + hashes) and must be written to a gitignored/host-only location, never committed.
- **HTTPS only in prod** via Caddy + Let's Encrypt for `APP_DOMAIN`, HTTP→HTTPS redirect (already in `caddy/Caddyfile`); README documents the one-line DNS A-record step.
- CI stays simple: lint + tests + frontend build (existing) + Docker image build (new). No deploy automation required.
- Tests run from `backend/` with the 5433 test DB and SHARED conftest fixtures (no local engine fixture). Pristine + ruff clean.

## Task 1: Purge endpoint (right to erasure)

**Files:** Extend `backend/app/routers/admin.py`, `schemas.py`; create `backend/tests/test_purge.py`.

**Interfaces:**
- `PurgeRequest{ confirm: str }`.
- POST `/api/admin/purge` (admin): require `body.confirm == "PURGE ALL CANDIDATE DATA"` exactly, else 400 "confirmation phrase did not match". In ONE transaction, delete (FK-safe order): `download_event` (all), `question` (all), `booking` (all), `candidate` (all); plus `audit_log` rows where `actor` is a candidate id (i.e. `actor != "admin"`). Then audit `data_purge` (actor="admin", detail = counts summary, no PII). Commit. Return `{deleted: {candidates, bookings, questions, download_events, audit_rows}}`.
- (Use the existing `current_admin` dep; `# noqa: B008`.)

- [ ] **Step 1: Failing tests** (`test_purge.py`, shared fixtures): seed an admin (via seed) + 2 candidates with bookings, questions, download_events, and candidate-actor audit rows (login etc.), plus an admin-actor audit row. Cases:
  - wrong/missing confirm phrase → 400, and NOTHING is deleted (counts unchanged).
  - correct phrase → 200; afterwards `candidate`/`booking`/`question`/`download_event` tables are EMPTY; candidate-actor audit rows are gone; the admin account + config rows + admin-actor audit rows REMAIN; a new `data_purge` audit row exists.
  - admin-only (non-admin → 401).

- [ ] **Step 3: Implement** with explicit deletes in FK-safe order (download_event → question → booking → candidate; then audit_log where actor != "admin"). Capture counts before deleting for the response/audit detail.

## Task 2: Admin config view/set

**Files:** Extend `backend/app/routers/admin.py`, `schemas.py`; create `backend/tests/test_config_admin.py`.

**Interfaces:**
- GET `/api/admin/config` (admin) → all config as `{key: value}` (include `prep_window_days`, `retention_date`, `qa_sla_text`, `display_timezone`, and any others present).
- `ConfigSet{ value: str | None }`.
- PUT `/api/admin/config/{key}` (admin) → set/update the config row for `key`. Validate by key:
  - `prep_window_days`: must parse as a positive int → else 422.
  - `retention_date`: must be null/empty (unset) OR an ISO date `YYYY-MM-DD` → else 422. (Stored as string; null clears it.)
  - `qa_sla_text`, `display_timezone`: free string (non-validated, but reject empty for display_timezone).
  - Only allow a known set of keys (reject unknown key → 400) to avoid arbitrary config writes.
  - Audit `config_update` (detail = the key only, not necessarily the value; never secrets). Return the updated `{key, value}`.

- [ ] **Step 1: Failing tests:** GET returns seeded config incl. prep_window_days=8 and retention_date null; PUT prep_window_days="10" → persists and GET reflects it; PUT prep_window_days="-1"/"abc" → 422; PUT retention_date="2026-12-31" → persists; PUT retention_date="" → clears (null); PUT retention_date="nonsense" → 422; PUT unknown key → 400; admin-only.
- [ ] **Step 2: RED.**
- [ ] **Step 3: Implement** with a small ALLOWED_KEYS set + per-key validators. Upsert the config row.
- [ ] **Step 4: GREEN** + ruff clean + full suite.

- [ ] **Step 1:** Write `scripts/backup.sh`: `pg_dump` the DB (using `PGPASSWORD`/`PGHOST=db`/`PGUSER`/`PGDATABASE` env) to `/backups/assessment-$(date +%Y%m%d-%H%M%S).sql.gz` (gzip), then prune dumps older than N days (e.g. `find /backups -name '*.sql.gz' -mtime +7 -delete`). `set -euo pipefail`, LF endings.
- [ ] **Step 2:** Add a `backup` service to `compose.yaml` using `postgres:16` (so `pg_dump` matches the server version), mounting `scripts/backup.sh` and a named `backups` volume at `/backups`, env wired to the db, `depends_on: db (healthy)`. Command: a small `sh -c` loop that runs `backup.sh` once a day (e.g. `while true; do /backup.sh; sleep 86400; done`) — boring host-cron-equivalent, no extra infra. Document that the operator can also run it on demand via `docker compose run --rm backup /backup.sh`.
